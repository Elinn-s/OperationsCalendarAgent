# PRD：门店通告系统化管理平台

**版本：** 0.1（草稿）  
**日期：** 2026-05-14  
**状态：** 待评审，部分需求待澄清（见末节）

---

## 1. 背景与痛点

当前通告以讯息（企业微信/邮件）形式流转，存在以下问题：

| 痛点 | 具体表现 |
|------|----------|
| 无唯一标识 | 通告无系统 ID，无法精确引用与追踪 |
| 无状态流转 | 不知道通告是否已发出、是否被收到、是否已过期 |
| 无提前预警 | 依赖人工记忆，频繁漏发或临期才想起来 |
| 责任不清 | 通告制定与发布责任不透明，出问题无法溯源 |
| 无历史检索 | 过往通告散落在各聊天记录，无法系统查阅 |

样例通告（`H_GO26_001`）中已存在规范化的字段：档案编号、拟稿/复核/审核三级签署、执行时间、适用范围、联系人——说明**线下规范已有雏形，缺的是系统化的承载工具**。

---

## 2. 产品目标

将通告全生命周期（计划 → 制定 → 发布 → 执行 → 归档）纳入系统管理，实现：

- **可管理**：每份通告有唯一 ID、状态、责任人
- **可追踪**：发布记录、确认记录有据可查
- **可预警**：关键节点提前提醒，不再靠人记

---

## 3. 目标用户

| 角色 | 描述 |
|------|------|
| **通告制定人**（拟稿） | 各部门负责起草通告的同事 |
| **通告审批人**（复核/审核） | 对通告内容负责的管理层 |
| **通告管理员** | 负责维护通告计划、预设提醒的运营人员 |
| **通告查阅人** | 需要查阅历史或当前有效通告的同事 |

---

## 4. 核心数据模型

基于样例通告提炼的通告实体字段：

```
通告（Notification）
├── 系统字段
│   ├── notification_id        系统自动生成的唯一 ID（UUID）
│   ├── status                 状态：草稿 / 待审批 / 已发布 / 执行中 / 已过期 / 已归档
│   └── created_at / updated_at
│
├── 从 PDF 自动提取（AI 解析）
│   ├── doc_ref                档案编号，如 H_GO26_001
│   ├── system_no              单号，如 PBTGFB202601160005
│   ├── department             发文部门，如 总务部
│   ├── drafter                拟稿人
│   ├── reviewer               复核人（可多人）
│   ├── approver               审核人
│   ├── title                  通告标题
│   ├── purpose                通告目的（摘要）
│   ├── target_scope           适用范围
│   ├── effective_start        执行开始时间
│   ├── effective_end          执行截止时间
│   ├── archive_until          电子存档截至
│   └── contacts               联系人列表（姓名、邮箱、分区）
│
├── 人工录入/确认
│   ├── owner                  当前责任人（系统内用户）
│   └── tags                   标签（如：春节、物料、包装）
│
└── 附件
    └── original_file          原始 PDF 文件
```

## 5. 功能需求

### F1：通告上传与关键信息自动提取

**用户故事：** 作为通告制定人，我上传一份完成的通告 PDF，系统自动识别关键字段，免去手动录入。

**功能细节：**
- 支持上传 PDF 格式通告文件
- 调用 LLM 解析以下字段（见第 4 节）：档案编号、发文部门、拟稿/复核/审核人、标题、目的摘要、适用范围、执行起止时间、联系人
- 提取结果展示给用户**人工确认/修正**，确认后保存
- 自动分配系统唯一 `notification_id`
- 若执行截止时间距今 ≤ 5 天，上传时即触发高优提醒

**待澄清：**
- [ ] 通告 PDF 是否有统一模板？提取准确率需实测后决定是否引入结构化表单兜底
- [ ] 档案编号（`H_GO26_001`）是否作为业务主键，还是另建系统 ID？

---

### F2：已知活动/通告计划预录入

**用户故事：** 作为通告管理员，我知道每年有哪些固定活动需要发通告（如春节、双 11），我想提前把计划录入系统，让系统在该发通告前自动提醒。

**功能细节：**
- 支持创建"通告计划"：填写活动名称、拟发通告内容、计划发布日期、责任人
- 可设置两级提醒：
  - **开始制作提醒日**：提示"需要开始起草此通告"
  - **发布提醒日**：提示"通告发布时间临近，请确认发布进度"
- 计划与实际通告可关联（录入后与上传的 PDF 绑定）
- 在通告提取页下方管理备忘录，支持粘贴文案后自动拆字段并人工确认

**待澄清：**
- [ ] 固定活动清单是否有现成来源（如 HR 行事历、营运年度计划）？
- [ ] 提醒时间节点 14 天 / 7 天是否合适，还是需要按通告类型灵活配置？

---

### F3：到期预警通知（邮件/企业微信）

**用户故事：** 作为责任人，我希望在通告即将到期前收到提醒，不需要自己盯着日历。

**功能细节：**
- 预警触发规则（初步，可配置）：
  - 通告执行截止前 **5 天**：提醒责任人"通告即将到期"
  - 通告执行截止前 **3 天**：升级提醒，同时抄送上级
  - 通告执行截止后 **1 天**：自动将状态改为"已过期"，记录入日志
- 通知渠道：
  - **邮件**（初期实现，SMTP 发送）
  - 企业微信 Webhook（后期扩展，待确认接入方式）
- 通知内容包含：通告标题、档案编号、截止日期、查看链接

**待澄清：**
- [ ] 责任人是否等于"拟稿人"，还是需要另设"执行负责人"字段？
- [ ] 是否需要接收人"已读确认"功能？

---

### F4：历史通告追踪与智能检索

**用户故事：** 作为查阅人，我想快速找到某个活动/部门/时间段的历史通告，或者用自然语言问"春节期间有哪些物料通告"。

**功能细节：**

**4a. 结构化检索**
- 按字段筛选：部门、状态、执行时间范围、标签、责任人
- 每条通告展示：档案编号、标题、状态、责任人、执行期、上传日期
- 支持下载原始 PDF

**4b. 自然语言检索（AI 问答）**
- 用户输入问题，如："2026 年有哪些关于包装物料的通告？"
- 系统检索历史通告向量库，返回相关通告列表及摘要
- 技术方案：PDF 文本 → 向量化（Embedding）→ 存入向量数据库 → RAG 检索

**4c. 审计日志**
- 每份通告记录完整操作历史：上传、确认、状态变更、提醒发送记录

**待澄清：**
- [ ] 历史通告数量量级？（影响向量库方案选型）
- [ ] 是否有权限管控需求？（某些通告只有特定部门可查）

### F7：自动提醒与升级（Automated Reminder & Escalation，待实现）

**目标：** 无需使用者逐一追踪，系统根据截止日期自动提醒、自动标记逾期，并在必要时升级通知。

**需求：**
- Deadline 前自动提醒（T-x days），如 T-7 / T-3 / T-1，可配置。
- Deadline 当日提醒。
- Overdue 自动标记为已逾期。
- Overdue 自动升级（escalation），如通知负责人上级、管理层或指定群组。
- 记录提醒和升级日志，避免重复发送。

**当前状态：**
- 已实现：页面看板显示临近截止、逾期未回执；应用启动时自动扫描 Deadline，支持 T-x / 当日 / 逾期升级邮件；逾期会自动标记为已逾期；提醒日志用于避免同日重复发送。
- 部分实现：可通过 `run_reminders.py` 接 Windows 任务计划实现无人值守；当前内网试用版仍依赖运行环境持续在线。
- 未实现：企业微信/Teams 集成、复杂升级链路、正式用户上级关系。

### F8：完整历史与争议追踪（已实现基础版）

**目标：** 保留完整历史，便于事后复盘、KPI/incentive 争议处理和责任追踪。

**必须记录：**
- 何时创建 / 发布通告
- 谁发布或操作
- 谁收到提醒 / 回执邮件
- 谁确认，何时确认
- 何时标记完成
- 何时自动逾期
- 何时触发提醒或升级
- 邮件发送成功/失败与错误原因

**当前状态：**
- 已实现：`audit_log`、`email_log`、`reminder_log`、`ack_recipients` 共同形成历史时间线。
- 已实现：通告详情页可查看“完整历史 / 事后追踪”。
- 待增强：正式登录后记录真实操作人；字段修改 diff 级别审计。

## 6A. 用户故事分析（当前探索版）

1. **As a HKM Head**：希望一眼看到高风险通告仍未完成，避免 KPI / incentive 受影响。  
   **含义：** 管理层视角不是看所有通告，而是看“高风险 + 未完成 + 逾期/即将逾期”。需要高影响范围、逾期项、未完成项聚合。

2. **As an Ops Lead**：希望清楚知道哪些 deadline 若未完成会影响下游人员。  
   **含义：** 营运负责人需要“我负责的项目 + 影响对象/范围 + deadline 风险”，重点是下游影响而非单条通告文本。

3. **As a Store Manager**：只想看到与我相关、需要我行动的通告。  
   **含义：** 门店视角必须过滤噪音，只呈现与门店/区域/岗位匹配的行动项，避免增加前线负担。

4. **As Central Ops**：希望不用人手查谁未回复或忘记执行。  
   **含义：** 系统必须自动追踪未回执、未完成、逾期和升级，减少人工 follow-up。

## 6B. 成功指标（Success Metrics）

- 通告漏发 / 迟发 incident 次数下降。
- Overdue item 能在 deadline 前被识别。
- 管理者 human follow-up 次数下降。
- 使用者对“责任清晰度”的满意度提升。

## 6C. 风险与考虑（Risks & Considerations）

- **Adoption**：避免增加前线负担，UX 必须简洁，门店端只看相关行动。
- **Scope creep**：需清晰区分 notification management 与深度营运系统，Phase 1 不扩展成完整营运平台。
- **Integration**：初期与既有工具并行，优先 Email，后续再接 Teams / 企业微信。

## 6D. Phase 1 Scope（Explore / Pilot）

- 定义 notification schema。
- 建立基本 dashboard + reminder flow。
- 小范围 HKM pilot，选择几类高风险通告先试。
- 验证字段识别、提醒、回执和历史追踪是否能降低人工跟进成本。
